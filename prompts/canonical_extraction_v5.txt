## How to apply the rubric: in-scope vs out-of-scope DS responsibilities

When evaluating a JD, FIRST identify which of the role's responsibilities fall into the IN-SCOPE list (data science work) vs the OUT-OF-SCOPE list (work outside data science). Then apply the per-score rubric above based on the responsibility distribution. The lists are tools to make the rubric application precise — they do NOT replace the per-score thresholds.

### IN-SCOPE DS responsibilities

These constitute "the data science work itself":

1. Stakeholder problem discovery — gathering requirements for DS-shaped problems with PM/business/research lead
2. Data exploration / EDA
3. ML model development (training, hyperparameter tuning, evaluation, feature engineering)
4. Statistical analysis (hypothesis testing, regression, inference, causal methods)
5. Experimentation (A/B test design, experiment analysis, causal inference)
6. Model deployment / monitoring / retraining (MLOps within DS scope — model serving, drift detection, retraining loops)
7. Communicating findings to stakeholders (presenting model results, explaining limitations and uncertainty)
8. Research direction setting (autonomous research, paper publication) — counts as in-scope when in a research-oriented role

### OUT-OF-SCOPE responsibilities

These constitute work outside the data science role:

1. Data engineering — ETL pipelines, data ingestion, data warehouse/lakehouse infrastructure beyond model serving needs
2. Dashboard / report development — Streamlit, Tableau, PowerBI, Looker beyond simple demos
3. Backend engineering — REST APIs, microservices NOT directly serving models
4. Frontend / full-stack web development
5. System / solution architecture, cloud architecture
6. DevOps / cloud infrastructure (CI/CD, IaC) beyond MLOps for model serving
7. Software engineering principles / SDLC ownership for non-ML systems
8. Domain-specific quantitative modeling that's a SEPARATE professional discipline — financial modeling for trading/risk, actuarial modeling, geophysical/seismic modeling, biostatistical clinical trial design — when these are the PRIMARY work, NOT when DS supports such activities. (Per the "Quant/financial-modeling roles → fit=3" note above, these typically land in fit=3.)
9. Project / people management for non-DS teams (manager of a software team, manager of a solutions team)

### Application reasoning

Internal reasoning loop the model should follow:

  Step 1: List the role's responsibilities from the JD
  Step 2: Tag each responsibility as in-scope (with item #) or out-of-scope (with item #)
  Step 3: Determine the proportion: predominantly in-scope, mixed, or predominantly out-of-scope
  Step 4: Apply the per-score rubric:
            - Predominantly in-scope, no notable out-of-scope → 5
            - Predominantly in-scope + ONE notable out-of-scope item (commonly item 2: dashboards) → 4
            - Significant out-of-scope alongside in-scope → 3
            - Predominantly out-of-scope with limited in-scope → 2
            - No data work at all → 1
  Step 5: Apply conservative default if score is borderline between 3 and 4 (drop to 3)
  Step 6: Apply quant/financial-modeling note (fit=3) if domain expertise is the primary work
