You are extracting canonical fields from a job description for deduplication. Return STRICT JSON matching the schema below. Do not include markdown fences or any text outside the JSON object.

=== CANONICAL COMPANY ===
Official short name only — NO legal suffixes (Inc, Ltd, Corp, Co., Canada Ltd, Inc., Limited, L.L.C., LP).
Examples:
  "Toronto-Dominion Bank Canada Ltd." → "TD Bank"
  "Shopify Inc." → "Shopify"
  "Royal Bank of Canada" → "RBC"
  "Lululemon Athletica Canada Inc." → "Lululemon"
  "Recruiting in Motion" → "Recruiting in Motion"
  "Outlier Ai" → "Outlier AI"

For small/regional companies, prefer the LONGER descriptive form to avoid
collisions with other companies sharing a common word:
  "Alquemy Search & Consulting" — keep ALL distinguishing words; do NOT
  truncate to "Alquemy Search" because that drops "& Consulting" which
  identifies the specific firm.
The Inc/Ltd-stripping rule applies ONLY to legal suffixes
(Inc, Ltd, Corp, Co., Limited, L.L.C., LP). It does NOT apply to descriptive
parts of the name (Search, Consulting, Group, Holdings, Partners, Advisory,
Solutions, Technologies — these may BE part of the canonical name).

=== CANONICAL SENIORITY ===
Pick EXACTLY ONE of: Junior | Mid | Senior | Staff | Principal | Lead | Manager | Director
Never invent a value outside this list.
Examples:
  "Sr. Software Engineer" → "Senior"
  "Software Engineer II" → "Mid"
  "Software Engineer I" → "Junior"
  "Tech Lead" → "Lead"
  "Engineering Manager" → "Manager"
  "Architect" → "Principal"
  "Staff Engineer" → "Staff"
  "VP of Engineering" → "Director"
  "Principal Data Scientist" → "Principal"
  "Data Scientist" (no qualifier) → "Mid"
  "Junior Data Analyst" → "Junior"
  "Head of Data" → "Director"

IGNORE HR classifications when extracting seniority:
  "Staff - Non Union" → administrative HR pay band, NOT IC seniority "Staff"
  "Pay Band X", "Job Family Y", "Grade Z", "Job Code N" → administrative; ignore
  "Excluded - Management" / "Excluded - Professional" → administrative; ignore
Extract seniority from the IC level title (Engineer I, Senior Engineer,
Principal Scientist, etc.) or, if not stated, default to "Mid".

=== CANONICAL LOCATION ===
Use ONLY these values — never return a province, country, or postal code:
  Vancouver | Toronto | Montreal | Calgary | Edmonton | Ottawa | Halifax
  Hybrid — Vancouver | Hybrid — Toronto | Hybrid — Montreal | Hybrid — Calgary | Hybrid — Edmonton | Hybrid — Ottawa | Hybrid — Halifax
  Remote — Canada | Remote — North America | Remote — Global
  Other

Metro-area aliasing (map suburbs → canonical city):
  Burnaby, Richmond, Surrey, Coquitlam, New Westminster, Delta, Langley, Abbotsford, Kelowna → "Vancouver"
  Mississauga, Brampton, North York, Etobicoke, Markham, Scarborough, GTA, Waterloo, Kitchener, Hamilton → "Toronto"
  Laval, Longueuil, Brossard, Greater Montreal → "Montreal"
  Any city not in the canonical city list → "Other"

Remote mapping:
  "Remote (Canada)" or "Remote - Canada" or "Work from home, Canada" → "Remote — Canada"
  "Remote (US and Canada)" or "Remote - North America" → "Remote — North America"
  "Remote (Global)" or "Remote — Worldwide" or "Remote (anywhere)" → "Remote — Global"
  "Hybrid - Vancouver, BC" → "Hybrid — Vancouver"
  "Hybrid - Toronto, ON" → "Hybrid — Toronto"
  Hybrid roles in suburb cities → use the canonical metro: "Hybrid - Burnaby" → "Hybrid — Vancouver"
  If the role has no explicit remote/hybrid/in-office signal → use the city name or "Other"

=== TEAM OR DEPARTMENT ===
Organisational unit only — the team or department the role reports into.
NOT role-level details (no project names, no specializations, no products).
If the JD does not explicitly name a team or department, return null.
Length is flexible — single-word org units like "Engineering" or "IT" are fine
when that's literally the department; multi-word org paths like "Advanced
Technology / Applied AI Research" are fine when that's the actual structure.
What matters is org-unit semantics, not word count.

Good: "Engineering", "IT", "Marketing Analytics", "RBC Borealis",
      "Quantitative Equity Team", "Advanced Technology / Applied AI Research"
Bad (role-level): "Customer LTV Modelling", "Pricing Algorithm", "Demand Forecasting"

If the team name is unclear or the JD only describes the role itself
(no organisational context), return null. Do NOT invent a team name.

=== TOP SKILLS ===
Up to 10 skills ordered by salience (most prominent first).
Use short canonical labels: "Python", "SQL", "PyTorch", "Spark", "dbt", "Kubernetes", "A/B testing", etc.

=== ROLE SUMMARY ===
3–4 neutral sentences summarising the role for deduplication embedding.
No marketing language ("exciting opportunity", "world-class team"), no salary figures, no company boilerplate.
Focus on: what the role does, what skills are central, what the team works on, seniority/scope.
